# 寻梦：作品案例拆解

## 产品定位

目标用户是醒来后只能记住零散画面、情绪和身体感受的人。普通日记要求用户把梦写完整，《寻梦》先允许碎片存在，再把碎片编织成可回看的体验。

## 用户问题与产品回应

- 梦消失得快：文字、语音和标签让记录更轻。
- 用户担心 AI 改写自己的梦：生成前先抽取意象、人物、地点和身体感受，关键事实保留可追溯。
- 单段文本缺乏体验：叙事拆成 2-4 个场景，图像、文字和氛围音共同推进。
- 多张图不连贯：统一 global_style、重复实体描述和 seed。
- 生成之后没有理由回来：梦境库、日历、详情、重新编织和情绪统计形成长期管理。
- 情绪需要被安放：用长按收梦仪式结束体验，而不是停在 AI 输出页。

## 用户旅程

注册或访客进入 -> 留下梦境碎片 -> 标记情绪 -> AI 解析与织梦 -> 场景图渐进生成 -> 沉浸回看 -> 收梦 -> 日历与梦境库再次访问。

## AI 机制

第一阶段提取用户事实和情绪，第二阶段生成标题、统一视觉风格、场景叙事、视觉提示词和结束回应。结构化 JSON 直接驱动场景 UI。原始碎片、用户情绪和生成结果分别保存。模型失败时保留输入并使用本地叙事和渐变图，不伪造真实生成。

## 技术架构

- Vue 前端：路由、Pinia 状态、语音输入、场景切换、分享卡和 PWA。
- FastAPI 后端：账号、会话、梦境 CRUD、生成任务和静态资源。
- SQLAlchemy 与 SQLite：用户、梦境、碎片、情绪、叙事和 Asset。
- 模型适配层：文本固定为 StepFun Chat；图片默认 StepFun Images，并保留可配置的 Volcano Ark / SeedDream 适配器。
- 图片持久化：供应商响应立即下载到 backend/data/assets，再由本地 /assets 提供。
- 并发编排：叙事先返回，场景图片后台渐进生成并逐张落库。

## 两分钟讲法

寻梦解决的是梦境记录的输入不完整和回看体验薄弱。我没有要求用户先写完整故事，而是保留碎片、情绪和意象，再用两阶段模型先解析事实、后编织场景。生成结果不是一段文本，而是驱动沉浸页、场景图、日历和收梦仪式的结构化状态。工程上我完成了前后端迁入、品牌改造、StepFun 默认链路、可切换火山生图、本地图像持久化和一键验收。

## 高频追问

- 如何避免 AI 篡改：先抽取事实，再生成；提示词和评测检查关键意象保真。
- 图片如何连续：global_style、重复实体描述和 seed 共同约束。
- 为什么图片要下载：供应商 URL 有时效，本地持久化才能长期回看。
- 如何处理失败：叙事和图片分别降级，原始输入始终保留。
- 下一步：异步任务队列、对象存储、跨梦视觉角色一致性和更系统的生成评测。
